# 生成式 EHR 预测：富语料、CoT 与 RL 方案

> 更新日期：2026-07-18  
> 本文在 Simple SFT 基础上增加疾病/患者富语料、诊断推理激活和 VERL-GRPO。  
> 主要复用 SIDReasoner 的 `data_Qwen3.py`、`sft_Qwen3.py`、`sft_reasoning_activation.py`、`evaluate_Qwen3_think.py` 和内嵌 `verl/`。  
> 第一版 RL **只使用最终疾病 SID 是否命中完整 GT 集合的结果奖励**；rollout 自由生成，不挂 SID Trie，不加入格式奖励或 reasoning 奖励。本文暂不估算富语料生成成本。

## 1. 目标与三阶段训练

最终任务仍是：给定患者历史 ICD visit，预测下一次 visit 的完整 ICD 疾病集合。

每个疾病具有唯一 SID：

```text
ICD9CM:4280 → <a_17><b_93><c_4>
```

整体流程：

```text
已处理 EHR + disease.item.json + disease.index.json
  │
  ├─ 富语料离线蒸馏
  │    ├─ 疾病级：结构化语义卡 → SID 叙事
  │    └─ 患者级：reasoning_path → integrated_narrative
  │
  ├─ Stage 1：SID–医疗语义深度对齐
  │    T1–T9 多任务 SFT
  │
  ├─ Stage 2：CoT 推理激活
  │    历史 SID → <think>reasoning</think> → 一个疾病 SID
  │
  ├─ Stage 3：VERL-GRPO
  │    一 visit 一 prompt；完整 GT 集合；仅 exact outcome reward
  │
  └─ Evaluation
       自由生成一条 reasoning
       固定 reasoning + constrained beam → 疾病 rank-list
```

Stage 1 让模型理解 SID 的疾病语义；Stage 2 教模型按“先推理、再回答”的协议生成；Stage 3 让模型自己 rollout reasoning，并用最终预测结果优化策略。

## 2. 前置数据

本阶段假设 [`2026-07-18-simpleSFT.md`](./2026-07-18-simpleSFT.md) 中的数据已经准备完成：

```text
data/EHR/<dataset>/
├── manifest/disease_manifest.json
├── index/<dataset>.item.json
├── index/<dataset>.index.json
├── info/<dataset>.info.tsv
├── code_level/{train,valid,test}.csv
└── visit_level/{train,valid,test}.jsonl
```

必须保证：

- `disease_id ↔ SID` 一一对应且无碰撞。
- 历史使用 `cond_hist[:-1]`，目标是当前 visit 完整 ICD 集合。
- MIMIC-IV 使用 `system:code_norm` 作为疾病 ID。
- 历史保留 visit 边界和时间顺序。
- code-level 行可按 `sample_id` 回聚为 visit-level 完整 GT。
- patient ID、visit ID 和绝对日期不进入 prompt 或外部教师请求。

## 3. 富语料目录与消费关系

推荐目录：

```text
data/EHR/<dataset>/rich_corpus/
├── disease_enriched.json
├── patient_enriched.visit.jsonl
├── reasoning_activation.code_level.csv
├── rejected_samples.jsonl
└── generation_manifest.json

data/EHR/<dataset>/rl/
├── train.parquet
└── valid.parquet
```

各文件的消费关系：

| 文件 | 粒度 | 消费阶段 | 用法 |
|---|---|---|---|
| `disease_enriched.json` | disease-level | Stage 1 T7 | SID 疾病富语义纯 LM |
| `patient_enriched.visit.jsonl` | visit-level | Stage 1 T8 | SID 病程叙事纯 LM |
| `reasoning_activation.code_level.csv` | code-level | Stage 2 | `<think>{reasoning_path}</think> + target_sid` |
| `train/valid.parquet` | visit-level | Stage 3 | 自由 rollout + 完整 GT 集合 reward |

SIDReasoner 的 Amazon 数据一行只有一个 next item，因此同一张 `integrated_narrative.csv` 可以被多阶段复用。EHR 是多标签任务，必须拆开三种视图：

- Stage 1 的患者叙事每个 visit 只训练一次。
- Stage 2 将同一 reasoning 与该 visit 的每个目标 SID 分别配对。
- Stage 3 回到一 visit 一行，用完整 GT 集合打分。

## 4. 富语料生成的共同原则

富语料不是让教师自由创作临床故事，而是把已有、可追溯信息改写成 SID 可学习的语义和序列文本。

必须遵守：

1. 先输出结构化中间结果，再生成最终叙事。
2. 医学事实必须有来源；缺少来源的字段留空，不允许猜测补全。
3. 外部教师只接收完成任务所需的最小字段。
4. 患者级输入不包含 patient ID、visit ID、绝对日期或其他身份字段。
5. 具体疾病实体在最终叙事和 reasoning 中使用完整 SID 指代。
6. reasoning 不等于真实临床因果链，也不能作为临床诊断意见。
7. 每次生成记录教师模型、prompt version、输入 hash 和质量标记，支持复现与断点续跑。

## 5. 疾病级富语料蒸馏

疾病级流程对应 SIDReasoner 的 Item-centric Semantic Enrichment。

### 5.1 阶段 A：结构化疾病语义卡

每个疾病调用一次教师。输入只使用疾病元数据和允许使用的知识源：

```text
disease_id
ICD 名称
父节点名称
ontology path
带 source_id 的医学知识片段
```

教师输出固定 JSON：

```json
{
  "definition": "...",
  "body_system": "...",
  "broader_category": "...",
  "clinical_context": ["..."],
  "related_concepts": ["..."],
  "source_ids": ["ICD9CM", "ONTOLOGY_X"]
}
```

该阶段可以出现疾病名称和 code，因为它是结构化中间层，不直接作为最终 SID 叙事训练文本。

### 5.2 阶段 B：SID 叙事整合

第二次教师调用接收原始元数据、结构化语义卡和完整 SID，输出一段自然文本：

```text
<a_17><b_93><c_4> belongs to ... . In this context,
<a_17><b_93><c_4> is associated with ... .
```

要求：

- 指代该具体疾病时只能使用完整 SID。
- 最终文本不出现疾病名称、ICD code 或别名。
- 允许描述有来源的 body system、broader category 和 clinical context。
- 不添加无来源的症状、治疗、预后或因果关系。
- SID 至少出现两次，防止富文本与 SID 对齐过弱。

### 5.3 `disease_enriched.json`

为了直接复用 `SidTextInterleaveDataset_v2`，保留其所需的 `llm_stage2` 字段：

```json
{
  "ICD9CM:4280": {
    "disease_id": "ICD9CM:4280",
    "sid": "<a_17><b_93><c_4>",
    "ontology_path": ["...", "..."],
    "source_provenance": [
      {"source_id": "ICD9CM", "field": "name"}
    ],
    "llm_stage1": {
      "definition": "...",
      "body_system": "...",
      "broader_category": "...",
      "clinical_context": ["..."],
      "related_concepts": ["..."],
      "source_ids": ["ICD9CM"]
    },
    "llm_stage2": "<a_17><b_93><c_4> belongs to ...",
    "teacher_model": "...",
    "prompt_version": "disease-rich-v1",
    "quality_flags": []
  }
}
```

`data_Qwen3.py` 中的 `SidTextInterleaveDataset_v2` 只读取 `llm_stage2`，将其作为纯文本做 `labels=input_ids` 的 next-token SFT。医疗实现可直接复用该类，或新增 `EhrDiseaseRichDataset` 做严格 schema 校验。

## 6. 患者级 reference-guided 蒸馏

患者级流程对应 SIDReasoner 的 User-centric Reasoning Enrichment。每个 **train visit 只生成一条 reasoning**，不按目标疾病重复调用教师。

教师可以看到完整 next-visit GT 集合作为内部 guidance，但最终 `reasoning_path` 必须表现得像只根据历史进行盲预测。该方法存在事后合理化风险，因此必须做严格泄漏检查。

### 6.1 阶段 A：Reasoning Generation

教师输入：

```text
历史 visit 的疾病名称、SID 和可用疾病描述
保留 <visit_start> / <visit_end> 的时间顺序
Reference next-visit：当前 visit 完整 GT 名称和 SID 集合
```

Reference 必须标记为仅供内部 guidance。输出 `reasoning_path`，要求：

- 只输出简洁的第一人称分析过程。
- 描述持续、复发、近期变化和同 visit 共现等纵向模式。
- 具体疾病只使用历史中出现过的 SID。
- `reasoning_sids ⊆ history_sids`。
- 不出现疾病名称、ICD code、患者标识或绝对时间。
- 不出现非历史 GT SID，也不提及 reference、ground truth 或 target diagnosis。
- 不直接点名预测某一个 next disease。
- 不把时间先后或共现写成临床因果。
- 以 `may`、`could` 等不确定性总结收尾。

成品示意：

```text
I observe that <a_...><b_...><c_...> persists across multiple visits,
while <a_...><b_...><c_...> appears more recently alongside it. This pattern
may indicate that related conditions could remain relevant in a subsequent visit.
```

该字段会在 Stage 2 原样放入 `<think>...</think>`。

### 6.2 阶段 B：Narrative Integration

第二次调用接收完整历史、阶段 A 的 `reasoning_path` 和 reference next-visit GT，生成 `integrated_narrative`。

要求：

1. 开头按原顺序复述完整历史 SID，并保留 visit 边界。
2. 自然融合 reasoning 中的纵向观察。
3. 具体疾病只使用 SID，不出现疾病名称或 ICD code。
4. 结果部分可以自然落到真实 next-visit reference SID 集合。
5. 不引入历史与 reference 集合之外的 SID。

关键区别：

- `reasoning_path` 严禁泄露非历史 GT SID，用于 Stage 2 的 `<think>`。
- `integrated_narrative` 可以包含 reference outcome，只在 Stage 1 作为富语料使用。
- Stage 2 prompt 和 RL prompt 都不能读入 `integrated_narrative`。

### 6.3 `patient_enriched.visit.jsonl`

```json
{
  "sample_id": "mimic3:train:000001",
  "split": "train",
  "history_visits_sid": [
    ["<a_8><b_21><c_77>"],
    ["<a_14><b_62><c_9>"]
  ],
  "reference_next_visit_sids": [
    "<a_17><b_93><c_4>",
    "<a_14><b_90><c_31>"
  ],
  "reasoning_path": "I observe that ...",
  "integrated_narrative": "<visit_start> ... <visit_end> ...",
  "teacher_model": "...",
  "reasoning_prompt_version": "patient-reason-v1",
  "integration_prompt_version": "patient-integrate-v1",
  "quality_flags": []
}
```

患者富语料只从 train split 生成。valid/test 不能送给教师或作为富语料训练。

## 7. 自动质检与失败隔离

每条语料至少检查：

- JSON/schema 和必填字段。
- SID 正则解析及合法 manifest 校验。
- 疾病级 `llm_stage2` 包含目标 SID，且不含疾病名称/code。
- `reasoning_sids ⊆ history_sids`。
- `reasoning_path` 不含非历史 reference SID、疾病名称或 reference 暗示。
- `integrated_narrative` 的 SID 集合是 `history_sids ∪ reference_sids` 的子集。
- `integrated_narrative` 完整保留历史 visit 顺序。
- patient ID、visit ID、绝对时间和其他身份信息不存在。
- `<think>`、`</think>` 等控制 token 已从教师原始输出中清理。
- 文本非空、长度合理、没有异常重复或模板复读。

硬规则失败的样本写入 `rejected_samples.jsonl`，保留失败原因和生成元数据，不得静默修补后进入训练。

`generation_manifest.json` 至少记录：

```json
{
  "schema_version": 1,
  "teacher_model": "...",
  "prompt_versions": {
    "disease": "disease-rich-v1",
    "reasoning": "patient-reason-v1",
    "integration": "patient-integrate-v1"
  },
  "source_hashes": {},
  "generated_counts": {},
  "rejected_counts": {},
  "created_at": "..."
}
```

生成脚本应支持按 `disease_id` / `sample_id` 断点续跑、幂等写入和定期落盘。

## 8. Stage 1：富语义深度对齐

Stage 1 从 Qwen3 base 开始，复用 `sft_Qwen3.py` 的模型加载、扩词表、assistant-only mask、`MultiEvalTrainer`、DDP 和 checkpoint 保存。

训练任务：

| 任务 | 内容 |
|---|---|
| T1 | 疾病文本 → SID |
| T2 | SID → 疾病文本 |
| T3 | SID 历史 → 目标疾病文本 |
| T4 | 文本历史 → 目标疾病文本 |
| T5 | SID 历史 → 目标 SID（主任务） |
| T6 | 文本历史 → 目标 SID |
| T7 | 疾病级 `llm_stage2` 富语料 |
| T8 | 患者级 `integrated_narrative` 富语料 |
| T9 | 可选的通用/医疗推理保持语料 |

### 8.1 对应 `data_Qwen3.py`

- T1/T2：复用 `SidItemFeatDataset` 的医疗版。
- T3–T6：复用 Simple SFT 中的 EHR Dataset。
- T7：参考 `SidTextInterleaveDataset_v2`，读取 `llm_stage2`。
- T8：参考 `SidTextInterleaveSequenceDataset`，但改读 visit-level JSONL 的 `integrated_narrative`，避免多目标展开后重复同一叙事。
- T9：可参考 `GeneralSFTReasonDataset`，但不是本方案硬依赖。

T7/T8 是纯语言建模：

```text
input_ids = tokenize(rich_text)
labels = input_ids
```

教师调用时的隐藏 reference、疾病名称和结构化中间信息不应拼入训练文本。

### 8.2 任务混合

第一版可沿用 `plan-reason-rl.md` 的初始比例：

```text
T5 核心预测             30%
T1–T2 双向映射          20%
T3/T4/T6 预测对齐       20%
T7 疾病级富语料         10%
T8 患者级富语料         10%
T9 通用/医疗推理        10%
```

如果暂不使用 T9，可将其 10% 优先分配给 T5。不要直接把所有 Dataset `ConcatDataset` 后按自然长度训练；EHR code-level 数据会支配梯度。应使用显式 sampler 或 HuggingFace dataset interleave，并记录实际采样比例。

### 8.3 Stage 1 输出与验收

输出：

```text
output_dir/<dataset>_stage1_sft/final_checkpoint/
```

至少验证：

- T5 next-SID loss 和 visit-level Recall@K。
- T1 text→SID exact match。
- T2 SID→text 规范化 exact match。
- tokenizer 保存/重载后 SID 与 visit token 仍为原子 token。
- 加入 T7/T8 后相对 T1–T6 Simple SFT 的增益。

## 9. Stage 2：CoT 推理激活

Stage 2 必须从 Stage 1 最佳 checkpoint 初始化，复用 `sft_reasoning_activation.py` 和 `ReasoningActivationDataset` 的基本结构。

训练协议：

```text
历史 SID → <think>患者级 reasoning</think> → 一个真实目标 SID
```

监督输出：

```text
<think>
{reasoning_path}
</think>

<a_x><b_y><c_z>
```

### 9.1 code-level 配对文件

同一 visit 的 reasoning 与全部 GT 分别配对：

```text
shared reasoning + SID_A
shared reasoning + SID_B
shared reasoning + SID_C
```

`reasoning_activation.code_level.csv` 最少包含：

```text
sample_id,history_sid_visits,ground_truth_sids,target_sid,reasoning_path,target_index
```

概念记录：

```json
{
  "sample_id": "mimic3:train:000001",
  "history_sid_visits": [["<a_...><b_...><c_...>"]],
  "ground_truth_sids": ["SID_A", "SID_B"],
  "target_sid": "SID_A",
  "reasoning_path": "I observe that ...",
  "target_index": 0
}
```

Stage 2 Dataset 只能把历史、`reasoning_path` 和当前 `target_sid` 放入训练样本。`reference_next_visit_sids` 和 `integrated_narrative` 不能进入 system/user prompt。

### 9.2 训练混合

建议：

```text
Reasoning activation       85%
T5 SID history → SID       10%
T1 text → SID              2.5%
T2 SID → text              2.5%
```

当前 `sft_reasoning_activation.py` 实际只构造 `ReasoningActivationDataset`。EHR 版需要增加 15% replay，并使用显式任务采样，降低 Stage 2 对 Stage 1 SID 对齐能力的遗忘。

### 9.3 masking 与截断

- prompt 全部 mask。
- `<think>`、reasoning、`</think>`、最终 SID 和 EOS 参与 loss。
- 显式使用 `mask_eos=False`，并可视化 labels 确认 chat template 后缀正确。
- 超长样本先删除最旧完整 visit，再缩短 reasoning。
- 不允许截断掉 `</think>`、最终 SID 或 SID 的中间层。

Stage 2 验收：thinking 输出格式稳定；reasoning 中的 SID 来自历史；no-thinking Recall 相对 Stage 1 不出现明显退化。

## 10. Stage 3：visit-level VERL-GRPO

SFT 可按单目标拆分，RL 必须回聚到完整 visit：

```text
一个 visit → 一个 prompt → n 个自由 rollout → 完整 GT 集合结果奖励
```

否则同一 visit 中生成另一个真实疾病时会被错误惩罚。

### 10.1 VERL parquet

```json
{
  "data_source": "ehr/mimic3_sid_cot",
  "prompt": [
    {
      "role": "system",
      "content": "Analyze the longitudinal SID history, reason, then output one disease SID."
    },
    {
      "role": "user",
      "content": "<visit_start> ... <visit_end>"
    }
  ],
  "ability": "ehr_diagnosis_prediction",
  "reward_model": {
    "style": "rule",
    "ground_truth": [
      "<a_17><b_93><c_4>",
      "<a_14><b_90><c_31>"
    ]
  },
  "extra_info": {
    "sample_id": "mimic3:train:000001",
    "ground_truth_sids": ["...", "..."],
    "split": "train"
  }
}
```

若当前 parquet/VERL 版本不能稳定保存嵌套 list，则把 `ground_truth` 写为 JSON 字符串，在 reward 中用 `json.loads` 解析，禁止使用 `eval`。

### 10.2 自由 rollout

输出协议：

```text
<think>
模型自由生成的推理
</think>

<a_x><b_y><c_z>
```

RL rollout 使用 VERL 原生 vLLM 自由采样：

- 不挂 SID Trie。
- 不使用 `prefix_allowed_tokens_fn`。
- reasoning 和最终答案都由模型自由生成。
- 非法 SID、缺少 `</think>` 或无法解析答案时结果 reward 为 0。
- SID 硬约束只在最终评测使用。

### 10.3 仅结果奖励

第一版总奖励只有：

```text
R = R_outcome
```

不使用：

- `R_valid` 格式/合法性奖励；
- `R_reason` reasoning 规则奖励；
- LLM-as-a-Judge；
- 过程奖励模型；
- SID prefix 部分奖励；
- 长尾权重或覆盖多样性奖励。

从最终 `</think>` 后解析唯一完整答案 SID：

```text
若 answer_sid ∈ ground_truth_sids：reward = 1.0
否则：reward = 0.0
```

无法解析、出现多个答案 SID 或输出不完整时也返回 0。这里的格式检查只用于确定是否成功提取一个结果，不提供额外 reward。

参考核心：

```python
import json
import re

SID_PATTERN = re.compile(r"(?:<[A-Za-z]+_\d+>)+")


def parse_answer_sid(solution: str):
    if "</think>" not in solution:
        return None
    answer_region = solution.rsplit("</think>", 1)[1].strip()
    matches = SID_PATTERN.findall(answer_region)
    if len(matches) != 1:
        return None
    return matches[0]


def ehr_outcome_reward(data_source, solution_str, ground_truth, extra_info=None):
    if isinstance(ground_truth, str):
        ground_truth = json.loads(ground_truth)
    answer_sid = parse_answer_sid(solution_str)
    return float(answer_sid is not None and answer_sid in set(ground_truth))
```

正式实现还应根据 `disease.index.json` 生成的 SID 正则/层数校验完整答案，但不能硬编码 `<a>/<b>/<c>` 或品类路径。

结果奖励稀疏，可能出现同一 prompt 的全部 rollout 都为 0，从而组内 advantage 为 0。第一版应先依赖 Stage 2 提供足够好的初始化，并监控：

- exact reward 均值；
- 有非零 reward 的 prompt 比例；
- 每组 reward 标准差；
- 无法解析答案比例。

若长期全零，先检查 Stage 2、增加 `rollout.n` 或缩小 smoke-test 数据范围；不要在基线实验中悄悄加入其他 shaping reward。

### 10.4 VERL 初始配置

参考 `RL_training_script.sh`：

```text
model                         Stage 2 final checkpoint
algorithm.adv_estimator       grpo
rollout engine                vLLM
rollout.n                     16
actor learning rate           5e-7
KL loss coefficient           0.001
actor/ref                     FSDP
max prompt length             1024
max response length           512
gradient checkpointing        true
custom reward                 ehr_outcome_reward.py
```

需要将 `custom_reward_function.path` 指向新的医疗结果奖励文件，并移除 SIDReasoner 中硬编码 Amazon `info` 路径的 reward。

正式训练前先跑小样本 smoke test，确认：parquet 可加载、同一 prompt 的 16 个 rollout 正确分组、reward 不全零、KL 有限、checkpoint 可保存并合并。

## 11. 推理评测

Thinking 评测复用 `evaluate_Qwen3_think.py` 的两阶段思路：

1. 用 vLLM 自由生成一次 reasoning，直到 `</think>`。
2. 固定该 reasoning，在答案起点挂合法疾病 SID trie。
3. 用 HuggingFace constrained beam search 生成 Top-K SID rank-list。
4. 去重并将 SID 还原为 disease ID / ICD code。

即：

```text
推理放开，最终答案锁死。
```

评测时建议同时比较：

| checkpoint | no-thinking | thinking |
|---|---:|---:|
| Stage 1 | ✓ | 可选 |
| Stage 2 | ✓ | ✓ |
| Stage 3 | ✓ | ✓ |

rank-list 和指标沿用 [`2026-07-18-simpleSFT.md`](./2026-07-18-simpleSFT.md)：

- Recall@10/20/30/40；
- oracle-cardinality weighted-F1；
- SID 合法率、重复率和候选不足 K 比例；
- reasoning SID grounded rate 与泄漏审计。

训练 rollout 不挂 trie，与评测答案阶段挂 trie并不矛盾：训练保留自由探索，评测保证提交给集合指标的候选全部来自合法疾病空间。

## 13. 最小验收顺序

1. **富语料 smoke test**：每类先生成少量样本，完成自动质检和人工审读。
2. **Stage 1**：比较 T1–T6 与加入 T7/T8 后的 SID 对齐和 Recall@K。
3. **Stage 2**：确认能稳定生成 `<think>...</think> + SID`，且 no-thinking 指标不明显退化。
4. **Reward 单测**：完整 GT 中任一 SID 命中都得 1，错误/非法/多答案得 0。
5. **VERL smoke test**：验证自由 rollout、GRPO 分组、reward 分布和 checkpoint 合并。
6. **Stage 3**：比较 Stage 2/3 的 thinking 与 no-thinking 集合指标。

只有在结果指标、SID 合法性和 reasoning 泄漏审计同时通过后，才报告 CoT/RL 的有效性；自然语言推理较长或较流畅本身不代表医学预测更正确。
