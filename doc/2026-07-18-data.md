# MIMIC-III / MIMIC-IV 数据说明与基本操作

> 更新日期：2026-07-18  
> 本文面向本项目的生成式 EHR 预测任务：使用患者历史 visit 中的 ICD 诊断，预测下一次 visit 的 ICD 疾病集合。  
> 本文统计均针对仓库中当前的数据文件重新计算；MIMIC-IV 按“编码歧义已经过滤并解决”处理，不展开清洗过程。

## 1. 数据文件

```text
SIDReasoner/mimic-dataset/
├── patient_time_mimic3_1/
│   ├── mimic3_box_dataset_1.pkl
│   ├── trainset.pt
│   ├── validset.pt
│   ├── testset.pt
│   
├── patient_time_mimic4_1/
│   ├── mimic4_box_dataset_1.pkl
│   ├── trainset.pt
│   ├── validset.pt
│   └── testset.pt
└── raw/
    └── icd9_icd10_combined.csv
```

两个 `*.pkl` 都是 PyHealth 的 `SampleEHRDataset` 对象：

- MIMIC-III：`dataset_name=MIMIC3Dataset`，任务名为 `diag_prediction_mimic3_fn`。
- MIMIC-IV：`dataset_name=MIMIC4Dataset`，任务名为 `diag_prediction_mimic4_fn`。
- `dataset.samples` 是 `list[dict]`，每条样本表示“以某次 visit 为预测目标”的一条患者记录。
- `trainset.pt`、`validset.pt`、`testset.pt` 保存的是 `dataset.samples` 的整数下标，而不是独立复制的数据。

加载 pickle 会执行其中记录的 Python 对象构造逻辑，因此只应加载可信来源的数据文件，并使用安装了 `torch` 和 `pyhealth` 的环境。本项目现有数据可在 `karpathy` 环境中读取。

## 2. 加载数据

### 2.1 同时加载 MIMIC-III 和 MIMIC-IV

```python
from pathlib import Path
import pickle
import torch

DATA_ROOT = Path("SIDReasoner/mimic-dataset")


def load_mimic(version: int):
    if version == 3:
        data_dir = DATA_ROOT / "patient_time_mimic3_1"
        pkl_name = "mimic3_box_dataset_1.pkl"
        target_key = "icd9_code"
    elif version == 4:
        data_dir = DATA_ROOT / "patient_time_mimic4_1"
        pkl_name = "mimic4_box_dataset_1.pkl"
        target_key = "icd_code"
    else:
        raise ValueError("version must be 3 or 4")

    with (data_dir / pkl_name).open("rb") as handle:
        dataset = pickle.load(handle)

    split_indices = {
        split: [
            int(index)
            for index in torch.load(
                data_dir / f"{split}set.pt",
                map_location="cpu",
                weights_only=False,
            )
        ]
        for split in ("train", "valid", "test")
    }
    return dataset, split_indices, target_key


mimic3, mimic3_splits, mimic3_target_key = load_mimic(3)
mimic4, mimic4_splits, mimic4_target_key = load_mimic(4)

print(len(mimic3.samples))       # 8,692
print(len(mimic4.samples))       # 247,890
print(mimic3_target_key)         # icd9_code
print(mimic4_target_key)         # icd_code
```

这里需要显式传入 `weights_only=False`，因为 split 文件保存的是普通 Python 索引对象，而不是模型权重。

### 2.2 使用 `Subset` 和 DataLoader

```python
from torch.utils.data import Subset
from pyhealth.datasets import get_dataloader

train_set = Subset(mimic3, mimic3_splits["train"])
valid_set = Subset(mimic3, mimic3_splits["valid"])
test_set = Subset(mimic3, mimic3_splits["test"])

train_loader = get_dataloader(train_set, batch_size=32, shuffle=True)
valid_loader = get_dataloader(valid_set, batch_size=32, shuffle=False)
test_loader = get_dataloader(test_set, batch_size=32, shuffle=False)

batch = next(iter(train_loader))
print(batch.keys())
```

生成 SID/SFT/RL 数据时通常直接遍历 `dataset.samples` 和 split 下标更方便；训练传统 PyHealth baseline 时再使用 `Subset` 和 `get_dataloader`。

## 3. 单条样本的字段

两个数据集的公共结构如下：

| 字段 | 类型 | 含义 | 本项目第一版是否使用 |
|---|---|---|---|
| `visit_id` | `str` | 当前目标 visit 的标识 | 仅用于内部追踪，不进入 prompt |
| `patient_id` | `str` | 患者标识 | 仅用于划分检查，不进入 prompt |
| `conditions` | `List[str]` | 原判别式任务的聚合标签 | 否 |
| `cond_hist` | `List[List[str]]` | 按时间排列的历史 ICD visit；最后一项固定为空列表 | **是** |
| `procedures` | `List[List[str]]` | 历史与当前 visit 的操作编码 | 第一版否 |
| `drugs` | `List[List[str]]` | 历史与当前 visit 的药物编码 | 第一版否 |
| `adm_time` | `List[str]` | 历史与当前 visit 的入院时间 | 第一版否 |

目标字段存在一个命名差异：

| 数据集 | 当前 visit ICD 目标字段 | 编码范围 |
|---|---|---|
| MIMIC-III | `icd9_code` | ICD-9-CM |
| MIMIC-IV | `icd_code` | 已完成版本判定的 ICD-9-CM / ICD-10-CM |

`conditions` 是原有判别式任务使用的标签：MIMIC-III 中是 CCS 聚合标签；MIMIC-IV 中可见 `PRG...`、`FAC...` 等聚合标签。生成式 ICD 预测不使用该字段，最终目标应分别读取 `icd9_code` 或 `icd_code`。

### 3.1 visit 对齐关系

```text
visit 0（历史）  visit 1（历史）  ...  visit T（当前）
────────────────────────────────────────────────────
cond_hist[0]     cond_hist[1]          cond_hist[-1] = []
procedures[0]    procedures[1]         procedures[-1]
drugs[0]         drugs[1]              drugs[-1]
adm_time[0]      adm_time[1]           adm_time[-1]

当前诊断目标：icd9_code（MIMIC-III）或 icd_code（MIMIC-IV）
```

重要约定：

1. `cond_hist[-1]` 恒为 `[]`，用于避免当前 visit 的诊断泄漏到历史输入。
2. 真正的历史诊断必须读取 `cond_hist[:-1]`。
3. `cond_hist`、`procedures`、`drugs`、`adm_time` 的外层长度一致，同一下标对应同一 visit。
4. 每条样本至少有一次历史 visit，因此 `len(cond_hist) >= 2`。
5. 当前数据的目标 ICD 已去重；一个 visit 中每个目标 code 只出现一次。
6. MIMIC 的时间经过匿名化处理，适合计算同一患者内部的相对间隔，不应解释为真实日历时间。

### 3.2 读取历史和目标

```python
def extract_history_and_target(sample: dict, target_key: str):
    assert sample["cond_hist"][-1] == []
    assert (
        len(sample["cond_hist"])
        == len(sample["procedures"])
        == len(sample["drugs"])
        == len(sample["adm_time"])
    )

    history_visits = [list(visit) for visit in sample["cond_hist"][:-1]]
    target_codes = list(dict.fromkeys(sample[target_key]))
    return history_visits, target_codes


sample = mimic3.samples[mimic3_splits["train"][0]]
history_visits, target_codes = extract_history_and_target(
    sample, mimic3_target_key
)
```

使用 `dict.fromkeys` 可以在保持原顺序的同时去重。当前数据已经无目标重复，但下游预处理仍建议保留该步骤并增加断言。

## 4. split 的使用与检查

当前 MIMIC-III 和 MIMIC-IV 都按患者隔离划分。使用数据前建议运行以下检查：

```python
from itertools import combinations


def validate_splits(samples, split_indices):
    split_sets = {
        split: set(indices) for split, indices in split_indices.items()
    }

    # 样本下标互斥，并完整覆盖 dataset.samples。
    assert set().union(*split_sets.values()) == set(range(len(samples)))
    for left, right in combinations(split_sets, 2):
        assert not (split_sets[left] & split_sets[right])

    # 患者集合互斥。
    patient_sets = {
        split: {str(samples[index]["patient_id"]) for index in indices}
        for split, indices in split_indices.items()
    }
    for left, right in combinations(patient_sets, 2):
        assert not (patient_sets[left] & patient_sets[right])

    return patient_sets
```

不要重新进行逐样本随机切分，否则同一患者的不同 visit 可能落入不同 split，造成患者级信息泄漏。

## 5. 从 ICD code 获得疾病文本

疾病元数据文件为：

```text
SIDReasoner/mimic-dataset/raw/icd9_icd10_combined.csv
```

主要字段：

| 字段 | 含义 |
|---|---|
| `system` | `ICD9CM` 或 `ICD10CM` |
| `code` | 带标准小数点的疾病编码，例如 `008.45` |
| `code_norm` | 去掉小数点后的规范化编码，例如 `00845` |
| `parent_code` | 带小数点的父级编码 |
| `parent_code_norm` | 去掉小数点后的父级编码 |
| `name` | 英文疾病名称 |

数据集中的 ICD 通常使用无小数点形式，因此推荐使用 `(system, code_norm)` 作为稳定键，而不是只用 `code` 或只用疾病名称。

### 5.1 构建疾病文本查找表

```python
from pathlib import Path
import pandas as pd

METADATA_PATH = Path(
    "SIDReasoner/mimic-dataset/raw/icd9_icd10_combined.csv"
)

metadata = pd.read_csv(METADATA_PATH, dtype=str).fillna("")
metadata = metadata.loc[
    metadata["system"].isin(["ICD9CM", "ICD10CM"]),
    [
        "system",
        "code",
        "code_norm",
        "parent_code",
        "parent_code_norm",
        "name",
    ],
].copy()

metadata["code_norm"] = metadata["code_norm"].str.replace(
    ".", "", regex=False
)

icd_to_record = {
    (row.system, row.code_norm): {
        "code": row.code,
        "code_norm": row.code_norm,
        "parent_code": row.parent_code,
        "parent_code_norm": row.parent_code_norm,
        "name": row.name,
    }
    for row in metadata.itertuples(index=False)
}
```

### 5.2 MIMIC-III：ICD-9-CM code → 文本

MIMIC-III 的 `icd9_code` 和 `cond_hist` 都是 ICD-9-CM，可直接固定 `system="ICD9CM"`：

```python
def normalize_icd(code: str) -> str:
    return str(code).strip().replace(".", "")


def mimic3_icd_to_text(code: str) -> str:
    key = ("ICD9CM", normalize_icd(code))
    if key not in icd_to_record:
        raise KeyError(f"Missing ICD-9-CM metadata: {code}")
    return icd_to_record[key]["name"]


print(mimic3_icd_to_text("00845"))
# Intestinal infection due to Clostridium difficile
```

当前 MIMIC-III 中历史与目标出现的 4,491 个不同 ICD 均能在元数据中找到文本，覆盖率为 100%。

### 5.3 MIMIC-IV：带系统信息的 code → 文本

本文假设 MIMIC-IV 的 ICD 版本已经完成判定。下游处理中应保留判定后的 `system`，再以 `(system, code_norm)` 查找：

```python
def mimic4_icd_to_text(system: str, code: str) -> str:
    key = (system, normalize_icd(code))
    if key not in icd_to_record:
        raise KeyError(f"Missing metadata: system={system}, code={code}")
    return icd_to_record[key]["name"]
```

建议疾病 manifest 使用带系统的稳定 ID：

```json
{
  "disease_id": "ICD10CM:E785",
  "system": "ICD10CM",
  "code_norm": "E785",
  "name": "Hyperlipidemia, unspecified",
  "parent_code_norm": "E78"
}
```

不要对元数据简单执行 `drop_duplicates("code_norm")` 后任取第一条；MIMIC-IV 的疾病主键和 SID 映射应使用 `system:code_norm`。

### 5.4 建议生成统一疾病 manifest

后续 SID tokenizer、SFT、RL 和评测都应读取同一份 manifest，而不是各自重新查 CSV。推荐结构：

```json
{
  "ICD9CM:4280": {
    "disease_id": "ICD9CM:4280",
    "system": "ICD9CM",
    "code_norm": "4280",
    "name": "Congestive heart failure, unspecified",
    "parent_code_norm": "428",
    "sid_tokens": ["<a_17>", "<b_93>", "<c_4>"],
    "sid": "<a_17><b_93><c_4>"
  }
}
```

最低验证要求：

- 历史与目标中的每个疾病 ID 都能找到文本。
- `disease_id ↔ SID` 一一对应，无碰撞。
- manifest 中没有空 `name`。
- train/valid/test 共用同一份只含疾病元数据的 manifest；不能向其中写入标签频次等 split 信息。

## 6. 当前数据规模

### 6.1 visit-level 规模

visit-level 指 `dataset.samples` 中的原始预测样本数；每条样本包含一段历史和当前 visit 的完整 ICD 集合。

| 数据集 | split | visit 样本数 | 患者数 | 占 visit 样本比例 |
|---|---:|---:|---:|---:|
| MIMIC-III | train | 6,141 | 3,814 | 70.7% |
| MIMIC-III | valid | 837 | 544 | 9.6% |
| MIMIC-III | test | 1,714 | 1,091 | 19.7% |
| **MIMIC-III** | **总计** | **8,692** | **5,449** | **100%** |
| MIMIC-IV | train | 173,557 | 55,361 | 70.0% |
| MIMIC-IV | valid | 25,227 | 7,902 | 10.2% |
| MIMIC-IV | test | 49,106 | 15,820 | 19.8% |
| **MIMIC-IV** | **总计** | **247,890** | **79,083** | **100%** |

患者数不能跨 split 相加后再去重；当前三个 split 的患者集合本身互斥，因此表中总数恰好等于三者之和。

### 6.2 历史长度和目标集合大小

| 数据集 | 指标 | 最小值 | 平均值 | 中位数 | 最大值 |
|---|---|---:|---:|---:|---:|
| MIMIC-III | 历史 visit 数 | 1 | 1.918 | 1 | 28 |
| MIMIC-III | 历史诊断条目数 | 1 | 26.058 | 16 | 434 |
| MIMIC-III | 历史不同诊断数 | 1 | 19.913 | 15 | 140 |
| MIMIC-III | 当前目标 ICD 数 | 1 | 15.212 | 14 | 39 |
| MIMIC-IV | 历史 visit 数 | 1 | 5.460 | 3 | 237 |
| MIMIC-IV | 历史诊断条目数 | 1 | 60.456 | 25 | 2,048 |
| MIMIC-IV | 历史不同诊断数 | 1 | 31.434 | 20 | 435 |
| MIMIC-IV | 当前目标 ICD 数 | 1 | 12.417 | 11 | 39 |

其中“历史诊断条目数”保留不同 visit 间的重复诊断；“历史不同诊断数”对一名患者当前样本的全部历史 code 取集合。

### 6.3 疾病词表规模

| 数据集 | 历史 ICD 数 | 目标 ICD 数 | 历史与目标交集 | 历史与目标并集 |
|---|---:|---:|---:|---:|
| MIMIC-III | 3,954 | 3,710 | 3,173 | **4,491** |
| MIMIC-IV | 20,942 | 21,537 | 19,306 | **23,173** |

SID 空间必须覆盖“历史与目标并集”，不能只为训练目标中出现的疾病建立 SID，否则历史序列或验证/测试目标可能无法编码。

相对于训练 split 的目标冷启动情况：

| 数据集 | valid 目标词表 | valid 中训练未见目标 | test 目标词表 | test 中训练未见目标 |
|---|---:|---:|---:|---:|
| MIMIC-III | 1,602 | 86 | 2,170 | 190 |
| MIMIC-IV | 10,813 | 564 | 13,582 | 1,105 |

这里“训练未见”指该疾病没有出现在 train 的历史或目标中。使用全数据疾病元数据构造 tokenizer/manifest 属于 transductive vocabulary，应在实验报告中明确声明；模型训练样本仍只能来自 train split。

### 6.4 code-level 展开规模

生成式 SFT 将一个当前 visit 的多 ICD 集合拆成多条单目标 completion：

```text
同一 history → target ICD A
同一 history → target ICD B
同一 history → target ICD C
```

展开后的记录数等于每条 visit 样本的 `len(target_codes)` 之和：

| 数据集 | train | valid | test | 总计 |
|---|---:|---:|---:|---:|
| MIMIC-III | 93,616 | 12,448 | 26,161 | **132,225** |
| MIMIC-IV | 2,157,153 | 315,452 | 605,508 | **3,078,113** |

这也是 `data-metric.html` 中“next-疾病预测记录”的统计口径。它不是患者数，也不是原始 visit 数。

## 7. 为生成式 EHR 构造基础记录

建议先构造患者级记录，再从患者级记录派生 code-level SFT 数据。不要只保存拆分后的单目标行，否则后续 RL 和集合评测难以恢复完整 GT。

### 7.1 患者级 visit 记录

```python
def build_visit_record(sample, target_key, split, sample_index):
    history_visits, target_codes = extract_history_and_target(
        sample, target_key
    )
    return {
        "sample_id": f"{split}:{sample_index}",
        "split": split,
        "history_icd_visits": history_visits,
        "target_icd_codes": target_codes,
    }
```

真实 `patient_id` 和 `visit_id` 可以保存在受控的本地审计映射中，但不应进入训练 prompt、外部教师请求或公开产物。

### 7.2 code-level SFT 记录

```python
def explode_targets(visit_record):
    return [
        {
            **visit_record,
            "target_index": target_index,
            "target_icd_code": target_code,
        }
        for target_index, target_code in enumerate(
            visit_record["target_icd_codes"]
        )
    ]
```

每条拆分记录必须继续保留：

- 稳定的 `sample_id`；
- 完整 `target_icd_codes`；
- 当前子目标 `target_icd_code`；
- `target_index`。

这样 Stage 1/2 可以按单疾病监督训练，而 Stage 3 RL 和评测可以按 `sample_id` 回聚为完整 visit 集合。

## 8. 最小数据质量检查

正式生成 SID 或训练数据前，至少检查：

```python
def validate_samples(samples, target_key):
    for sample in samples:
        assert sample["cond_hist"]
        assert sample["cond_hist"][-1] == []
        assert len(sample[target_key]) > 0
        assert len(sample[target_key]) == len(set(sample[target_key]))
        assert (
            len(sample["cond_hist"])
            == len(sample["procedures"])
            == len(sample["drugs"])
            == len(sample["adm_time"])
        )
```

还应验证：

1. split 下标互斥且覆盖全部样本。
2. train/valid/test 患者集合无交叉。
3. ICD 文本映射覆盖历史和目标并集。
4. visit 顺序保持不变；只能在训练取样时动态打乱同一 visit 内的疾病顺序。
5. valid/test 使用确定性疾病排序，保证评测可复现。
6. 截断历史时按完整 visit 从最旧端删除，不能截断到 SID 中间。
7. SFT 拆分记录能够按 `sample_id` 无损回聚到患者级 GT 集合。

## 9. 口径总结

- 模型输入：`cond_hist[:-1]`，即当前 visit 之前的 ICD 历史。
- 模型目标：MIMIC-III 的 `icd9_code` 或 MIMIC-IV 的 `icd_code`。
- 第一版不使用 `conditions`、procedures、drugs 和时间。
- SFT/推理激活阶段可把一个 visit 拆成多条单疾病 completion。
- RL 和最终评测必须保留一个 visit 的完整真实疾病集合。
- MIMIC-III 疾病主键可写为 `ICD9CM:code_norm`；MIMIC-IV 必须保留 `system:code_norm`。
- “8,692 / 247,890”是 visit-level 样本数；“132,225 / 3,078,113”是 code-level 展开记录数。
